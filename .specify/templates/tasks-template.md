# 任务：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档  
**前置条件**：`plan.md`（必需）、`spec.md`（必需，用于用户故事），以及可选的
`research.md`、`data-model.md`、`contracts/`

**测试说明**：下面示例包含测试任务。自动化测试除非规格明确要求，否则可以不写；
但只要功能影响布局、导航、文章渲染或媒体展示，就必须在任务中加入响应式检查、
无障碍检查、链接/媒体验证和基础性能验证。

**组织方式**：任务按用户故事分组，以确保每个故事都能独立实现、独立验证。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可以并行执行（不同文件、无直接依赖）
- **[Story]**：对应的用户故事编号（例如 `US1`、`US2`、`US3`）
- 描述中必须写出明确文件路径

## 路径约定

- **单体项目**：仓库根目录下使用 `src/`、`tests/`
- **静态个人站点/博客**：使用 `index.html`、`pages/`、`posts/`、`assets/`、
  `styles/`、`scripts/`，必要时可有 `data/` 或 `qa/`
- **Web 应用**：使用 `backend/src/`、`frontend/src/`
- **移动端**：使用 `api/src/`、`ios/src/` 或 `android/src/`
- 下方示例默认按单体项目展示，生成任务时必须根据 `plan.md` 的真实结构调整

## 依赖与执行顺序

### Phase 依赖

- **准备阶段（Phase 1）**：无依赖，可立即开始
- **基础能力（Phase 2）**：依赖准备阶段完成，且会阻塞所有用户故事
- **用户故事阶段（Phase 3+）**：依赖基础能力完成
  - 若团队有人手，可并行推进多个故事
  - 若单人开发，可按 P1 → P2 → P3 顺序推进
- **最终阶段**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：基础能力完成后即可开始，不依赖其他故事
- **用户故事 2（P2）**：基础能力完成后即可开始，可与 US1 集成但必须保持可独立验证
- **用户故事 3（P3）**：基础能力完成后即可开始，可与 US1/US2 集成但必须保持可独立验证

### 单个用户故事内部顺序

- 如果有测试，必须先写并确认失败，再开始实现
- 先模型，再服务，再页面/接口
- 先核心能力，再集成细节
- 当前故事验证完成后，再进入下一个优先级故事

### 并行机会

- 所有标记 `[P]` 的准备任务可以并行
- 所有标记 `[P]` 的基础任务可以并行
- 基础能力完成后，不同用户故事可并行
- 同一故事中标记 `[P]` 的测试可并行
- 同一故事中标记 `[P]` 的模型可并行
- 不同故事可由不同成员并行推进

## 实施策略

### 先做 MVP（只完成用户故事 1）

1. 完成 Phase 1：准备阶段
2. 完成 Phase 2：基础能力
3. 完成 Phase 3：用户故事 1
4. **停止并验证**：确认用户故事 1 可独立工作
5. 如已满足目标，可先发布或演示

### 增量交付

1. 完成准备阶段 + 基础能力
2. 实现用户故事 1，并验证后发布/演示
3. 实现用户故事 2，并验证后发布/演示
4. 实现用户故事 3，并验证后发布/演示
5. 每个故事都应在不破坏前一故事的前提下增加价值

### 多人并行策略

当有多个开发者时：

1. 团队一起完成准备阶段与基础能力
2. 基础能力完成后：
   - 开发者 A：负责用户故事 1
   - 开发者 B：负责用户故事 2
   - 开发者 C：负责用户故事 3
3. 各故事独立完成并逐步集成

## 备注

- `[P]` 表示不同文件、无直接依赖，可并行
- `[Story]` 用于将任务映射回具体用户故事
- 每个用户故事都应该可独立完成、独立验证
- 开始实现前先确认测试失败（如适用）
- 发布前必须完成响应式、无障碍、性能和内容完整性检查
- 建议按任务或逻辑小组提交变更
- 到达检查点时应暂停并单独验证当前故事
- 避免出现：任务描述模糊、同文件高冲突、破坏独立性的跨故事强依赖
