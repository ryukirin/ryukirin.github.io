# ryukirin.github.io 宪章

## 核心原则

### I. 内容服务于个人表达
每项变更都必须直接强化以下至少一个目标：个人介绍、日常博客或技术分享。
任何无法让作者的身份、思考、作品或写作更清晰的页面、区块和功能，都不得
进入项目范围。理由：这个站点的价值来自持续建立辨识度与信任感，而不是堆积
泛用页面。

### II. 简洁而有质感
界面必须简洁、从容、克制，同时不能显得空洞、草率或只有骨架。每次面向用户
的改动都必须明确其排版、留白、配色和层级设计；装饰性效果只有在确实提升
气质、辨识度或可扫描性时才可加入。理由：本项目追求的“简洁”是精致和自信，
而不是功能缺席或视觉贫乏。

### III. 静态优先交付
站点必须保持 GitHub Pages 兼容的静态交付能力。新能力必须优先采用静态生成、
构建期数据或渐进增强；任何需要服务端运行时、数据库或常驻后端的方案，都
必须在实施计划的复杂度记录中书面说明必要性。理由：静态优先能让发布流程
保持可靠、低成本、低维护。

### IV. 易读、快速、无障碍
首页、列表页和文章页都必须优先保障移动端与桌面端的阅读舒适度、语义化
HTML、键盘可访问性以及轻量性能。核心内容必须在无 JavaScript 的情况下仍然
保持可理解；任何较重的素材或交互都必须先证明其读者价值。理由：个人站点
只有在清晰、快速、容易消费时，才更可能形成复访。

### V. 可持续写作与中文优先
内容结构、展示方式和维护流程都必须服务于单人长期更新。项目中的 README、
宪章、计划、规格、任务、清单等文档必须以中文撰写；框架名、API 名、协议名
和其他专有名词可以保留原文，但首次出现时必须放在中文语境中说明。复用布局、
命名规范、元数据约定和内容结构必须优先于一次性页面逻辑。每篇技术文章都
必须保留可读的代码格式、有效链接和清晰的发布日期信息。理由：项目应当在
时间有限时仍然可以稳定产出，并保持文档与内容表达一致。

## 体验标准

- 站点必须具备清晰的信息架构，至少覆盖首页身份展示、个人介绍、博客发现与
  文章详情等核心阅读路径。
- 每个功能提案都必须说明目标读者、主要行为或阅读路径，以及它为内容表达
  带来的具体价值。
- 字体层级、间距系统、颜色与可复用交互模式必须集中管理，不能按页面各自
  临时定义。
- 博客内容必须支持长文阅读、代码块、列表、引用和媒体展示，且不能破坏移动
  端布局或页面性能。
- 自定义域名、SEO 和社交分享卡片可以继续增强，但不得破坏静态优先的运作
  模式。

## 工作流与质量门槛

- 实施计划必须通过宪章检查，至少覆盖内容契合度、视觉方向、静态交付方式、
  响应式表现与维护成本。
- 规格文档必须包含可独立验证的用户故事；当涉及界面或内容改动时，还必须写
  清阅读质量、理解效率或发现效率的可衡量结果。
- 任务文档必须在功能影响布局、导航或文章渲染时，纳入内容建模、响应式验证、
  无障碍检查和轻量性能检查。
- README、plan、spec、tasks、checklist、constitution 等项目文档必须使用中文；
  如需保留英文术语，应以中文说明为主。
- 在合并或发布前，必须至少完成一次现代移动端宽度与桌面端宽度的人工检查，
  并确认关键链接、资源和元数据有效。
- 任何新依赖或复杂交互都必须以书面方式说明其读者价值与必要性。

## 治理

本宪章高于仓库中的临时偏好与口头约定。任何修订都必须同时满足以下条件：
更新本文件、同步相关模板或说明文档，并以语义化版本号记录变更级别和理由。

合规检查在规划阶段以及重要发布或合并前都必须执行。版本策略如下：删除或
重定义核心原则时升级 MAJOR；新增原则或实质性扩展治理要求时升级 MINOR；
仅作措辞澄清或非语义修饰时升级 PATCH。
